# Facebook Brand Assets Guide

來源：`sources/Facebook_Brand_Assets_Guide_2019.pdf`
版本：1.3
日期：2019-04
研究視角：Design Infrastructure Researcher

## Summary

這份文件規範 Facebook 品牌資產、商標使用、permission flow、f logo、thumb icon、Find Us On Facebook badge、Reactions 與 UI templates。這章先把 Facebook brand assets 定義成受控商標資產，避免第三方以不一致或具誤導性的方式呈現。

最上層的原則是先建立判斷標準，再進入視覺、語氣或應用規格。一致性、可辨識、避免混淆、避免暗示 partnership/sponsorship/endorsement 是核心治理原則。採「approved source + allowed contexts + prohibited modifications + permission gate」框架。

執行上，這份 guideline 透過 Welcome and General Guidelines、General usage and talking about Facebook、Requesting Permission、Facebook Brand Assets 等章節，把抽象原則落到可操作的資產、版面、語氣、流程或治理規則。平台品牌 guideline 應優先處理 trademark confusion 與 endorsement risk，再處理美學一致。

## Welcome and General Guidelines

1. **Chapter Purpose**：這章先把 Facebook brand assets 定義成受控商標資產，避免第三方以不一致或具誤導性的方式呈現。
2. **Definition**：它定義指南適用者、資產來源、一般禁止事項、文案提及規則與申請 permission 的情境。
3. **Relationship**：後續每個 asset 的細則都建立在此章的商標保護、資產來源與非背書原則上。
4. **Design Execution**：作者用 do/don’t、permission checklist 與使用情境分類，將 legal risk 轉成設計操作。
5. **Principles**：一致性、可辨識、避免混淆、避免暗示 partnership/sponsorship/endorsement 是核心治理原則。
6. **Frameworks**：採「approved source + allowed contexts + prohibited modifications + permission gate」框架。
7. **Patterns**：把 brand guideline 寫成第三方 compliance manual，而不是單純視覺風格書。
8. **Rules**：AI 可檢查：只能使用 Brand Resource Center 資產；Facebook wordmark 不可用於代表 presence；不得讓 Facebook 成為最突出特徵；不得改色、變形、旋轉、加效果或重畫資產；Facebook 一詞不可複數化、動詞化或縮寫。
9. **Reusable Knowledge**：平台品牌 guideline 應優先處理 trademark confusion 與 endorsement risk，再處理美學一致。
10. **Abstraction**：`Legal and Trademark`、`Governance`、`Content Design`。
11. **Visual Evidence**：章節以 do/don’t 清單與情境文字為主，視覺證據較少；其功能是法律與流程證據。

### General usage and talking about Facebook

1. **Chapter Purpose**：讓第三方知道何時能提 Facebook、如何提、以及不可把商標變成普通語言。
2. **Definition**：定義文案中 Facebook 的拼寫、大小寫、字級與語法；不授權任何 logo 使用。
3. **Relationship**：它補足 asset rules，處理沒有圖形資產時的文字商標風險。
4. **Design Execution**：用同字級同樣式、首字母大寫、不得以 logo 取代文字等規則維持商標完整性。
5. **Principles**：商標需被視為 proper noun，而非一般動詞、情緒詞或 icon。
6. **Frameworks**：採「text mention compliance」框架。
7. **Patterns**：品牌提及規則可以從視覺 guideline 延伸到 copy guideline。
8. **Rules**：AI 可檢查：除網址外需寫作 Facebook；字體大小與 surrounding text 一致；不得寫成 facebook、Facebooks、facebooked、FB；不得用 logo/icon 取代單字。
9. **Reusable Knowledge**：商標文字規則適合轉為 NLP lint。
10. **Abstraction**：`Voice and Tone`、`Legal and Trademark`、`AI-Checkable Rules`。
11. **Visual Evidence**：頁面含簡短例示與 do/don’t 欄位，作為文案審核證據。

### Requesting Permission

1. **Chapter Purpose**：把高風險使用情境導入審核流程，避免未授權曝光造成法律或品牌誤解。
2. **Definition**：定義哪些 use cases 需要 permission、提交哪些材料、何時由 Ad Policy team 審查。
3. **Relationship**：它與所有 asset 章節平行，是使用前的流程 gate。
4. **Design Execution**：要求提交 final version、commercial/film/program description、script、translation。
5. **Principles**：越接近大眾媒體、商業廣告、包裝與劇情敘事，審查需求越高。
6. **Frameworks**：採「use context risk classification + submission checklist」框架。
7. **Patterns**：permission section 將品牌治理流程化，降低設計師自行判斷法律風險。
8. **Rules**：AI 可檢查：TV/online marketing、books/plays/TV/film scripts、print packaging、news/commentary 需 permission；request 必須附最終版本、描述、script 與非英語 translation。
9. **Reusable Knowledge**：品牌資產 guideline 應內建使用情境風險表與審核材料清單。
10. **Abstraction**：`Governance`、`Legal and Trademark`、`Operations`。
11. **Visual Evidence**：此節以流程文字為主，視覺證據功能是 permission checklist。

## Facebook Brand Assets

1. **Chapter Purpose**：把常見第三方使用的 Facebook 資產逐一分流，避免 logo、thumb、badge、Reactions 被混用。
2. **Definition**：定義 f logo、social channel icon grouping、thumb icon、Find Us On Facebook badge 與 Reactions 的用途與限制。
3. **Relationship**：它把 general rules 轉成 asset-specific rules，並與 UI templates 章節銜接。
4. **Design Execution**：每個 asset 都用 purpose、clear space、shape/proportion、size、color 與 do/don’t 組成固定模板。
5. **Principles**：資產必須服務「引導到 Facebook presence」或「呈現 Facebook product context」，不得成為第三方創意主角。
6. **Frameworks**：採「asset purpose + allowed CTA + integrity constraints + prominence control」框架。
7. **Patterns**：每個 logo 類 asset 都用相同治理欄位，讓審核規則可模組化。
8. **Rules**：AI 可檢查：f logo 可用於 Facebook presence、Facebook content 或 product integration；thumb icon 必須搭配清楚 CTA 且不可與其他 Facebook symbols 並列；badge online 必須直接連到 Facebook presence；Reactions 不可單獨拆開或改造。
9. **Reusable Knowledge**：平台資產庫應將每個 asset 綁定單一任務與禁用情境。
10. **Abstraction**：`Logo System`、`UI Components`、`Legal and Trademark`。
11. **Visual Evidence**：章節使用 clear-space diagrams、do/don’t examples 與 UI templates；功能是比例、警示與規定證據。

### f Logo, Thumb Icon, and Badge

1. **Chapter Purpose**：讓第三方能正確導流到自己的 Facebook presence，而不是使用 corporate wordmark 或重製 icon。
2. **Definition**：定義 f logo、thumb icon 與 Find Us On Facebook badge 的角色、CTA、link、clear space、color 與 prominence。
3. **Relationship**：這些 asset 是外部 marketing 最常用入口，因此比 Reactions 更偏導流與商標識別。
4. **Design Execution**：用 clear space insulation、shift key proportional scaling、approved colors、direct link 要求維持完整性。
5. **Principles**：導流資產可以出現，但不能主導第三方 creative，也不能造成 Facebook endorsement。
6. **Frameworks**：採「presence promotion asset」框架。
7. **Patterns**：每個 asset 都必須和明確 action 或 context 綁定，防止漂浮裝飾化。
8. **Rules**：AI 可檢查：f logo 與 badge 使用時 online link 應導向 Page/Profile/Group/Event；f logo 與其他 social icons 並列時需等尺寸；thumb icon 只能搭配 Like/Install/For use with Facebook 類 CTA；顏色僅可用下載檔提供的 blue/grey/white/black 或技術限制下黑白。
9. **Reusable Knowledge**：CTA 與 link target 可成為 logo 使用的必要 metadata。
10. **Abstraction**：`Logo System`、`Interaction Design`、`AI-Checkable Rules`。
11. **Visual Evidence**：do/don’t 圖顯示正確導流、錯誤改色、錯誤比例與錯誤實體物件化。

### Reactions

1. **Chapter Purpose**：防止 Reactions 被當作 emoji、裝飾物或可自由拆分的情緒 icon。
2. **Definition**：定義 Like/Love/Haha/Wow/Sad/Angry collection、UI context、Newsfeed templates 與 Reactions Dock。
3. **Relationship**：它介於 brand asset 與 product UI 之間，因此規則比 logo 更重視產品語境。
4. **Design Execution**：要求 Reactions 以完整 collection 或 Facebook UI 內的真實互動方式呈現。
5. **Principles**：產品符號不可脫離產品語境，不可被創意再語義化。
6. **Frameworks**：採「product behaviour fidelity」框架。
7. **Patterns**：當 brand asset 來自 UI 元件時，規範需包含功能語境，而非只含外觀。
8. **Rules**：AI 可檢查：不得修改 Reactions 顏色、比例、形狀；不得單獨突出一個 Reaction 作為 icon；不得動畫化或做成實體物件；floating Reactions 僅能出現在 Facebook Live 或 Facebook products；若在 UI 內呈現需遵守 UI guidelines。
9. **Reusable Knowledge**：UI-derived assets 必須被當作「行為片段」而不是獨立圖形。
10. **Abstraction**：`UI Components`、`Product Language`、`Content Safety`。
11. **Visual Evidence**：Newsfeed、Reactions Dock 與 do/don’t 頁提供產品語境、裁切與完整 collection 的視覺證據。

## User Interface Templates

1. **Chapter Purpose**：讓第三方能展示 Facebook product UI，但不扭曲產品、隱私或平台關係。
2. **Definition**：定義 Profile/News Feed templates、自訂 copy/image、device context、cropping、font matching 與 privacy requirements。
3. **Relationship**：它承接 Reactions 的 UI context，擴展到完整螢幕模板使用。
4. **Design Execution**：要求 UI 放在相關 device 內，底部可裁切但不可裁上方或側邊，字體需對應 OS。
5. **Principles**：產品畫面必須像真實 Facebook，且在第三方 marketing 中保持 secondary role。
6. **Frameworks**：採「device-framed product representation」框架。
7. **Patterns**：可自訂內容，但不可改 UI 形態；這是 template-based localization 的治理模式。
8. **Rules**：AI 可檢查：Facebook UI 不得浮在頁面上；不得超過 100% file size；mobile UI 不得裁切 top/sides；iOS 用 San Francisco、Android 用 Roboto、Web 用 Helvetica；不得使用未授權 PII、第三方 logo、copyrighted content。
9. **Reusable Knowledge**：產品 UI guideline 需同時檢查畫面完整性、裝置語境與資料權利。
10. **Abstraction**：`UI Templates`、`Privacy`、`Legal and Trademark`。
11. **Visual Evidence**：device mockup 與 crop diagrams 直接提供可檢查的 UI framing 證據。

## FAQs

1. **Chapter Purpose**：用問答補強常見錯誤，降低使用者在商標、permission、merchandise 與 verb usage 上的誤判。
2. **Definition**：定義品牌資產規則背後原因、違規處理、permission、merchandise、assets source、verb usage 與 media usage。
3. **Relationship**：它回收前面章節的規則，轉成決策支援。
4. **Design Execution**：作者將 legal rationale 寫成可掃描問答，便於非設計人員查詢。
5. **Principles**：規定指南需要回答「為什麼」，不只列出「不可做」。
6. **Frameworks**：採「policy rationale + common scenario answer」框架。
7. **Patterns**：FAQ 是降低審核摩擦的知識庫入口。
8. **Rules**：AI 可檢查：Facebook assets 不可用於 merchandise；只能用 Brand Resource Center 官方資產；Facebook 不可作動詞；book/film/play 使用 logo 或 image 需書面 permission。
9. **Reusable Knowledge**：FAQ 可作為 rules engine 的 exception handling 與 user education layer。
10. **Abstraction**：`Knowledge Architecture`、`Governance`、`Legal and Trademark`。
11. **Visual Evidence**：此章幾乎全為文字問答，功能是政策推理證據而非視覺範例。
