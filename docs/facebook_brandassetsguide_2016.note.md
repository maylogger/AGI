# Facebook Brand Assets Guide

來源：`sources/facebook_brandassetsguide_2016.pdf`
版本：1.1
日期：2016-06
研究視角：Design Infrastructure Researcher

## Summary

這份 2016 版 Facebook Brand Assets Guide 規範一般商標使用、merchandising、permission、logos/badges、Like Button Icon、Thumb Icon、Reactions、Screens 與 FAQs。用 mission 與使用對象建立資產治理的正當性，讓第三方知道這不是設計偏好，而是品牌信任與商標保護。

最上層的原則是先建立判斷標準，再進入視覺、語氣或應用規格。品牌資產必須支援 Facebook 被正確認知，而非被第三方挪用成一般圖形。採「mission rationale + user scope + asset source」框架。

執行上，這份 guideline 透過 Welcome、General Guidelines、General Do’s / Don’ts and talking about Facebook、Merchandizing and requesting permission 等章節，把抽象原則落到可操作的資產、版面、語氣、流程或治理規則。品牌資產指南可把 mission 作為治理理由，而非只是放在首頁裝飾。

## Welcome

1. **Chapter Purpose**：用 mission 與使用對象建立資產治理的正當性，讓第三方知道這不是設計偏好，而是品牌信任與商標保護。
2. **Definition**：定義 Facebook mission、指南適用於 marketing/advertising 使用者，以及 Brand Resource Center 是完整資產與 permission 入口。
3. **Relationship**：這章是 General Guidelines 與 Brand Assets 的策略前言。
4. **Design Execution**：作者用「sharing/open and connected」敘事連接 logo 使用與人際連結產品價值。
5. **Principles**：品牌資產必須支援 Facebook 被正確認知，而非被第三方挪用成一般圖形。
6. **Frameworks**：採「mission rationale + user scope + asset source」框架。
7. **Patterns**：在法律式 guideline 前先說明品牌任務，可降低使用者對限制的抗拒。
8. **Rules**：AI 可檢查：資產來源應指向 facebookbrand.com；第三方 marketing material 需符合本 guide 與 Brand Resource Center。
9. **Reusable Knowledge**：品牌資產指南可把 mission 作為治理理由，而非只是放在首頁裝飾。
10. **Abstraction**：`Brand Strategy`、`Governance`、`Knowledge Architecture`。
11. **Visual Evidence**：本章文字為主，視覺證據功能是 mission framing。

## General Guidelines

1. **Chapter Purpose**：把商標風險、使用範圍、permission 與 merchandise 禁止事項明文化，防止外部使用者誤解授權邊界。
2. **Definition**：定義一般 do/don’t、Talking about Facebook、Merchandizing、Requesting Permission。
3. **Relationship**：它支撐後續 f logo、Like Button Icon、Thumb Icon、Badge、Reactions 與 Screens 的具體規則。
4. **Design Execution**：用清單化 do/don’t 與 permission 資料清單將法律語言轉為操作流程。
5. **Principles**：不可混淆、不可背書暗示、不可讓 Facebook 成為第三方材料最突出特徵、不可任意商品化。
6. **Frameworks**：採「asset integrity + trademark language + risk contexts + request flow」框架。
7. **Patterns**：把 permission 放在 guideline 早段，讓高風險使用先被流程攔截。
8. **Rules**：AI 可檢查：不得使用模仿 Facebook 的商標/域名/logo；不得將 Facebook 與第三方名稱或 generic terms 結合成新標誌；不得用於 pornography、illegal activities 或違反 Terms 的材料；Facebook 不可作動詞、複數或縮寫。
9. **Reusable Knowledge**：品牌治理應同時檢查視覺、語言、情境與媒體風險。
10. **Abstraction**：`Legal and Trademark`、`Content Safety`、`Governance`。
11. **Visual Evidence**：do/don’t 頁面以文字清單為主要證據，適合轉為審核規則。

### Merchandizing and requesting permission

1. **Chapter Purpose**：明確阻止將 Facebook IP 商品化，並把 TV/film 這類高曝光場景導入人工審核。
2. **Definition**：定義 merchandise 禁止與 TV/film permission request 的資料需求。
3. **Relationship**：它補上一般 logo 規則無法處理的授權和商業化邊界。
4. **Design Execution**：作者直接列出 clothing、hats、mugs 等不可授權品類，並要求 final version、screenshot、synopsis/script、translations。
5. **Principles**：平台標誌可以導流，但不能變成可販售商品的裝飾資產。
6. **Frameworks**：採「commercialization ban + media permission gate」框架。
7. **Patterns**：把商品化與媒體劇情使用視為不同風險類別。
8. **Rules**：AI 可檢查：不得在 clothing/hats/mugs 等 merchandise 使用 Facebook trademarks/logos/IP；TV/film 使用需透過 facebookbrand.com/requests；request 需包含最終呈現與腳本或 synopsis。
9. **Reusable Knowledge**：品牌指南應將「promotion use」與「merchandise use」明確分離。
10. **Abstraction**：`Legal and Trademark`、`Operations`、`Compliance`。
11. **Visual Evidence**：本節視覺少，主要證據是禁止品類與申請材料清單。

## Brand Assets

1. **Chapter Purpose**：讓每個 Facebook asset 都有明確任務，避免使用者把不同資產互相替代。
2. **Definition**：定義 Logos & Badges、Reactions、Screens 三組資產系統。
3. **Relationship**：它把 general rules 下沉到各 asset 的 use case、clear space、size、color、proportion 與 misuse。
4. **Design Execution**：作者以一致格式說明每個 asset 的用途、構造安全距、比例維持、顏色與 do/don’t。
5. **Principles**：資產的任務比形式更重要；每個圖形都必須保留產品語境或導流語境。
6. **Frameworks**：採「asset-specific permissible use」框架。
7. **Patterns**：不同 asset 共用治理欄位，形成可擴充品牌資產知識模型。
8. **Rules**：AI 可檢查：f logo 主要用於 presence；Like Button Icon 不可 online 或 Facebook Ads 使用；Thumb Icon 不可和其他 Facebook symbols 並列；Badge online 需直接連到 Facebook presence；Reactions 不可拆散或改造；Screens 需置於 device context。
9. **Reusable Knowledge**：品牌資產庫需要記錄每個 asset 的 allowed medium、required CTA、forbidden substitutions 與 permission status。
10. **Abstraction**：`Logo System`、`UI Components`、`Product Language`。
11. **Visual Evidence**：clear space 圖、logo 比例、do/don’t 與 screen mockup 提供比例與規定證據。

### Logos and Badges

1. **Chapter Purpose**：治理最常被外部使用的導流資產，避免 wordmark、f logo、Like Button、Thumb Icon、Badge 角色混淆。
2. **Definition**：定義 f logo、Like Button Icon、Thumb Icon、Find Us On Facebook Badge 的使用邊界。
3. **Relationship**：這是 Brand Assets 章中最偏 marketing activation 的部分。
4. **Design Execution**：f logo 與 badge 用於 presence，Like Button Icon 用於非功能性 print/broadcast/on-screen presentation，Thumb Icon 用於 like 概念但必須搭配 CTA。
5. **Principles**：導流資產不可被改造為裝飾圖案，也不可取代產品內真實互動元件。
6. **Frameworks**：採「presence / non-functional like / CTA thumb / badge」分類框架。
7. **Patterns**：每個圖示都綁定媒體與行動，避免成為自由 icon。
8. **Rules**：AI 可檢查：Like Button Icon 不得用於 online、online ads 或 Facebook Ads；Like Button/Thumb Icon 不得取代文字 like；Thumb Icon 必須搭配清楚 written CTA；f logo 不得成為最突出元素；所有 logo/badge 不得改色、變形、重畫、動畫化或實體物件化。
9. **Reusable Knowledge**：當 UI icon 被拿到 marketing 場景時，需標記為 functional 或 non-functional。
10. **Abstraction**：`Logo System`、`Interaction Design`、`Legal and Trademark`。
11. **Visual Evidence**：各 asset 頁面的 clear space 與 do/don’t 圖為主要視覺證據。

### Reactions

1. **Chapter Purpose**：保護 Reactions 作為產品互動集合的語境，避免被外部創意拆成一般 emoji。
2. **Definition**：定義 Like、Love、Haha、Wow、Sad、Angry collection、UI 使用、相關訊息與禁止事項。
3. **Relationship**：Reactions 連接 logo asset 與 Screens，因為它必須依附 Facebook UI 或完整 collection。
4. **Design Execution**：允許在 UI 中呈現，或在 UI 外搭配 relevant messaging 但仍保持完整 collection。
5. **Principles**：產品情緒符號不可單獨化、商品化或替代化。
6. **Frameworks**：採「collection integrity + contextual messaging」框架。
7. **Patterns**：完整 collection 是避免單一符號被過度語義化的治理模式。
8. **Rules**：AI 可檢查：Reactions 必須以完整 collection 顯示；不得替換、增加、改色、改比例、動畫化、實體化；TV & Film 使用需 prior written permission；不得使用非 facebookbrand.com assets。
9. **Reusable Knowledge**：情緒 icon guideline 應檢查「是否完整集合」與「是否保持原產品語境」。
10. **Abstraction**：`Product Language`、`UI Components`、`Content Safety`。
11. **Visual Evidence**：Newsfeed、Dock 與 highlighted Reaction 範例說明可接受的產品語境與訊息搭配。

### Screens

1. **Chapter Purpose**：允許第三方呈現 Facebook product screens，但防止 UI 被修改、誤導或侵犯資料權利。
2. **Definition**：定義 screen templates、Profile、Pages、Newsfeed、device framing、cropping、font 與 custom content。
3. **Relationship**：Screens 是 Reactions 的完整 UI 容器，也是 permission 與 privacy 規則的高風險場景。
4. **Design Execution**：要求 mobile/desktop device context、100% file size 上限、top/sides 不裁切、OS 對應字體。
5. **Principles**：產品畫面應真實、次要、在裝置中呈現，且自訂內容需有權利。
6. **Frameworks**：採「accurate UI reproduction + rights-cleared customization」框架。
7. **Patterns**：可自訂內容層，但不可改變 UI 結構層。
8. **Rules**：AI 可檢查：screens 不得大於 100% file size；不得裁切 mobile top/sides；iOS 用 San Francisco、Android 用 Roboto、Web 用 Helvetica；不得 blur/redact/annotate；不得使用未授權 PII、第三方商標或 copyrighted content。
9. **Reusable Knowledge**：UI template guideline 必須把設計完整性與資料授權列為同等規則。
10. **Abstraction**：`UI Templates`、`Privacy`、`Legal and Trademark`。
11. **Visual Evidence**：device mockup、Profile/Pages/Newsfeed screens 與 cropping diagram 是檢查依據。

## FAQs

1. **Chapter Purpose**：集中回答常見規定問題，降低品牌團隊反覆審查同類問題的負擔。
2. **Definition**：定義 trademark rationale、misuse enforcement、permission、merchandise、assets source、verb usage、TV/film 與 online ads。
3. **Relationship**：它把前面規則轉成可查詢的決策知識。
4. **Design Execution**：以短問答把規則與原因連結，讓非設計者也能理解。
5. **Principles**：規則需要被解釋，才能被長期遵守。
6. **Frameworks**：採「common question + policy answer + escalation path」框架。
7. **Patterns**：FAQ 可作為 guideline 的 support layer。
8. **Rules**：AI 可檢查：Facebook assets 不可用於 merchandise；只能用官方 Brand Resource Center 資產；Facebook 不可作動詞；film/book/play 使用 logo/image 需 written permission。
9. **Reusable Knowledge**：FAQ 內容適合轉入 rules engine 的例外與教育訊息。
10. **Abstraction**：`Knowledge Architecture`、`Governance`、`Compliance`。
11. **Visual Evidence**：文字問答為主，功能是政策證據與常見情境映射。
