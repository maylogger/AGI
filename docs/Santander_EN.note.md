# Basic Guidelines for Santander

來源：`sources/Santander_EN.PDF`
日期：2020
研究視角：Design Infrastructure Researcher

## Summary

Santander 2020 基礎品牌指南，提供 logotype、版本選擇、黑白版、留白、最小尺寸、cobranding、final artwork nomenclature 與 primary color palette。明確告知使用者這是 Santander 品牌實作的 basic guideline，任務是提供必要視覺資源與 final artwork，而不是完整品牌哲學書。

最上層的原則是先建立判斷標準，再進入視覺、語氣或應用規格。基礎 guideline 應專注於最高風險元素：Logo 與色彩。brand resources -> final artwork -> implementation。

執行上，這份 guideline 透過 Introduction、Logotype、Meaning and Versions、Clear Space and Minimum Size 等章節，把抽象原則落到可操作的資產、版面、語氣、流程或治理規則。不是每份 guideline 都需龐大；若目標是執行落地，文件應縮到關鍵決策。

## Introduction

1. **Chapter Purpose**：明確告知使用者這是 Santander 品牌實作的 basic guideline，任務是提供必要視覺資源與 final artwork，而不是完整品牌哲學書。
2. **Definition**：定義文件範圍為 logotype、color palette 與可用 artwork；不處理完整語氣、攝影或版面系統。
3. **Relationship**：開場將讀者導向「拿正確素材並正確使用」的操作模式，支撐後續 nomenclature。
4. **Design Execution**：以簡短說明降低進入成本，讓文件像品牌資產使用說明。
5. **Principles**：基礎 guideline 應專注於最高風險元素：Logo 與色彩。
6. **Frameworks**：brand resources -> final artwork -> implementation。
7. **Patterns**：短版 guideline 可採「必要資源 + 選擇規則 + 檔名系統」結構。
8. **Rules**：AI 可檢查準則：若任務是 Santander 基礎識別實作，應優先查找官方 final artwork，不應自行重繪 Logo。
9. **Reusable Knowledge**：不是每份 guideline 都需龐大；若目標是執行落地，文件應縮到關鍵決策。
10. **Abstraction**：`Operations`、`Visual Identity`。
11. **Visual Evidence**：開場頁視覺證據很少，功能是範圍界定。

## Logotype

1. **Chapter Purpose**：保護 Santander 企業標誌的可讀性、現代感與跨觸點一致性。
2. **Definition**：本章定義 logotype 意義、main/secondary/symbol/black-white 版本、clear space、minimum size、cobranding 與 final artwork file naming。
3. **Relationship**：是整份文件核心，Color Palette 則作為 Logo 與品牌環境的色彩基礎。
4. **Design Execution**：作者先解釋 flame、oval platform、red 的象徵，再立刻轉成版本選擇、尺寸與檔名規則。
5. **Principles**：不同尺寸與媒介需要不同版本，但所有版本都必須來自官方 artwork。
6. **Frameworks**：使用情境 -> 版本 -> condition -> color mode -> format。
7. **Patterns**：大型企業可用 logo version taxonomy 與 file nomenclature 讓素材管理機器化。
8. **Rules**：AI 可檢查準則：main version 應優先用於所有 communication；secondary version 僅尺寸限制導致 main version 不夠清楚時使用；symbol 僅小空間且 main/secondary 不可用時使用；無法印紅色時才用黑白 Logo。
9. **Reusable Knowledge**：Logo guideline 應提供「優先順序」，而不是只列全部版本。
10. **Abstraction**：`Logo System`、`Design Tokens`、`AI-Checkable Rules`。
11. **Visual Evidence**：章節使用正負版、symbol、尺寸、grid 與檔名示例，承擔比較、比例與資產治理功能。

### Meaning and Versions

1. **Chapter Purpose**：讓 Logo 版本選擇不只是技術決定，也延續品牌目的與現代化敘事。
2. **Definition**：定義 flame 表示 light、heat、humanity，oval platform 表示穩固，red 表示 energy、strength、bravery、command；並定義 main、secondary、symbol 與 black/white。
3. **Relationship**：品牌意義支撐為何 main version 應最大化使用，版本規則則支撐實際應用。
4. **Design Execution**：用範例直接展示 positive/negative 與版本差異。
5. **Principles**：Logo 的情感象徵與可讀性同等重要。
6. **Frameworks**：brand meaning -> version priority -> technical exception。
7. **Patterns**：版本分層應包含 preferred、fallback、extreme fallback。
8. **Rules**：AI 可檢查準則：credit cards 到大格式 communication 優先 main version；sponsorship 等尺寸受限可 secondary；app pictogram 或促銷小物等極小空間可 symbol。
9. **Reusable Knowledge**：Logo symbol 可獨立使用時必須明確列出限制情境。
10. **Abstraction**：`Visual Identity`、`Logo System`。
11. **Visual Evidence**：正負版與黑白版範例展示不同背景與印刷限制。

### Clear Space and Minimum Size

1. **Chapter Purpose**：確保 Logo 在大小與周邊干擾條件下保持清楚。
2. **Definition**：定義 preferred/minimum clear space、symbol security space 與各版本 minimum size。
3. **Relationship**：承接版本選擇，為每個版本設定物理使用底線。
4. **Design Execution**：main/secondary 使用 flame symbol width 作為理想留白，symbol 使用 28x28 modular grid。
5. **Principles**：留白應從 Logo 自身比例推出，才能跨尺寸縮放。
6. **Frameworks**：Logo type -> clear space unit -> minimum size。
7. **Patterns**：symbol 以 grid 管理，wordmark 以核心圖形寬度管理。
8. **Rules**：AI 可檢查準則：main version 最小 25px/3mm；secondary 50px/5mm；symbol 16px/3mm；main/secondary 理想 clear space 為 flame width，最低為半個 flame width；symbol security space 為 28x28 grid 中上左右 5 modules、下方 6 modules，最低上左右 3、下方 4。
9. **Reusable Knowledge**：同一 Logo family 中，不同版本可有不同測量邏輯。
10. **Abstraction**：`Layout System`、`Logo System`、`AI-Checkable Rules`。
11. **Visual Evidence**：以格線、尺寸數值與清楚標示呈現可測量證據。

### Cobranding

1. **Chapter Purpose**：在合作品牌並列時維持 Santander 的比例、距離與專業感。
2. **Definition**：定義 Santander 與 partner brand lockup 的 size ratio、grid、distance 與 dividing line。
3. **Relationship**：將 Logo 規則延伸到外部品牌關係。
4. **Design Execution**：以 grid 控制比例，在兩 Logo 間加入 1pt dividing line。
5. **Principles**：合作不應讓任一方失衡，比例必須被系統化。
6. **Frameworks**：lockup version -> modules -> distance -> separator height。
7. **Patterns**：cobranding 常用 separator line 與 module spacing 解決權重問題。
8. **Rules**：AI 可檢查準則：兩 Logo 間距為 4 modules；中央加入 1pt line；break line 高度依所選 lockup version 高度變化。
9. **Reusable Knowledge**：聯名規範應提供比例與距離，不應只說「等比例」。
10. **Abstraction**：`Brand Architecture`、`Layout System`。
11. **Visual Evidence**：以 lockup main、secondary、symbol 與 examples 展示不同情境。

### Final Artworks Nomenclature

1. **Chapter Purpose**：讓使用者能從檔名直接判斷是否選對版本、條件、色彩模式與格式。
2. **Definition**：定義命名結構 Final Artwork + Santander + Version + Condition + Color mode + Format。
3. **Relationship**：把前述版本規則轉成資產管理語法。
4. **Design Execution**：用 AF_SANTANDER_PV_POS_CMYK.EPS 示範可解析檔名。
5. **Principles**：檔名應是品牌治理的 metadata，而不是任意命名。
6. **Frameworks**：PV/SV/SY + POS/NEG + CMYK/PMS/RGB/BW + AI/EPS/JPG/PNG/PDF。
7. **Patterns**：以縮寫碼建立可排序、可搜尋、可驗證的 asset taxonomy。
8. **Rules**：AI 可檢查準則：檔名應包含 version code、condition、color mode 與 format；PV=Preferred version、SV=Secondary version、SY=Symbol；POS/NEG 表示正負版；CMYK/RGB/BW/PMS 表示色彩模式。
9. **Reusable Knowledge**：品牌資產命名可作為自動化檢查與 DAM 管理基礎。
10. **Abstraction**：`Operations`、`Design Tokens`、`Knowledge Architecture`。
11. **Visual Evidence**：檔名示例與欄位拆解是主要證據。

## Color Palette

1. **Chapter Purpose**：讓 Santander 色彩在所有 touch point 中保持平衡、現代與可執行。
2. **Definition**：定義 White、Red、Sky、Black 的 PMS、HEX、RGB、CMYK、RAL，以及建議色彩使用比例。
3. **Relationship**：支撐 Logo 與整體視覺環境，延伸 red 的品牌情緒。
4. **Design Execution**：用百分比建議控制白、紅、天空藍與黑的視覺重量。
5. **Principles**：紅色仍是核心，但需透過白色、藍色與黑色建立光感、簡潔與科技感。
6. **Frameworks**：primary color token + usage percentage + medium values。
7. **Patterns**：色彩 guideline 不只給色值，也給大致比例，能避免主色過度使用。
8. **Rules**：AI 可檢查準則：Red HEX #EC0000；White #FFFFFF；Sky #DEEDF2；Black #000000；建議色彩比例為 White 50%、Red 40%、Sky 5%、Black 5%。
9. **Reusable Knowledge**：色彩比例可轉成品牌一致性的自動化檢查基準。
10. **Abstraction**：`Color System`、`Design Tokens`。
11. **Visual Evidence**：使用色票、百分比與跨媒介色值作為 token 證據。

## Contact

1. **Chapter Purpose**：提供疑問處理路徑，避免使用者在不確定時自行判斷。
2. **Definition**：定義若有 guideline 疑問，應聯絡 Banco Santander brand team。
3. **Relationship**：結束文件並補足規則外情境。
4. **Design Execution**：以簡短頁面提醒溝通窗口。
5. **Principles**：基礎 guideline 的邊界之外需要品牌團隊決策。
6. **Frameworks**：uncertainty -> brand team contact。
7. **Patterns**：短版 guideline 可用 contact page 處理例外與疑義。
8. **Rules**：AI 可檢查準則：若 Logo 版本、色彩或聯名規則無法判定，應要求聯絡 brand team，不自行創建例外。
9. **Reusable Knowledge**：任何機器可讀品牌規則也需要人類 governance owner。
10. **Abstraction**：`Governance`、`Operations`。
11. **Visual Evidence**：文字提醒為主，功能是流程證據。
